- **Verification rule: visual inspection required for user-facing output.** For any change affecting user-facing output (emails, UI strings, push notifications, generated reports), verification must include visual inspection of the actually-rendered output, not just database state or API response. Auth-log routing analysis, "PATCH returned 200," and database state checks verify *behavior*, not *rendering*. Origin: 2026-05-14 OTP-cutover session. The Phase 1 template-content PATCH appeared to succeed by every database-state and API-routing signal — PATCH returned 200 with our values echoed back, auth logs showed correct action routing through `user_confirmation_requested`, the diagnostic Edge Function showed clean `email_confirmed_at` / `last_sign_in_at` / `auth.sessions` state — but the rendered email body was the Supabase default magic-link template with `{{ .ConfirmationURL }}`. Caught only when Buzz did the visual `+alias` test against a real email client. Lesson: end-to-end-from-user's-eye is the load-bearing check; every other signal is supplementary. Build at least one visual checkpoint into every fix that affects rendered output, BEFORE declaring fix-complete.

- **GSD hooks fail on Windows when Claude Code routes commands through bash.** Every GSD hook in `~/.claude/settings.json` is registered with PowerShell call-operator syntax (`& "C:/Program Files/nodejs/node.exe" "<hook>.js"` or `& bash "<hook>.sh"`). Claude Code invokes hook commands via `/usr/bin/bash -c`, where leading `&` is the background-process operator, not the PowerShell call operator. Result: every hook invocation fails with `syntax error near unexpected token '&'` before the hook script can run. PreToolUse variants block their tool calls (PreToolUse:Write|Edit hooks `gsd-prompt-guard.js`, `gsd-read-guard.js`, `gsd-workflow-guard.js`). PostToolUse variants fail silently (`gsd-context-monitor.js`, `gsd-read-injection-scanner.js`, `gsd-phase-boundary.sh`). The bug applies to ALL GSD hooks registered: `SessionStart` (`gsd-check-update.js`, `gsd-session-state.sh`), all `PreToolUse` matchers (`Write|Edit` × 3, `Bash` × 1), all `PostToolUse` matchers (`Bash|Edit|Write|MultiEdit|Agent|Task`, `Read`, `Write|Edit`), and the `statusLine` command (`gsd-statusline.js`). Workaround applied 2026-05-14: 5 most-impactful hooks disabled via empty `"hooks": []` arrays in `settings.json`; original committed to `settings.json.bak`; restored at session end so future sessions start with the broken-but-original config (avoids silently losing the GSD machinery without realizing it). Permanent fix: GSD-side change to either drop the `& ` prefix and invoke node/bash directly by path, or route through `powershell.exe -c "..."` / `cmd.exe /c "..."` so the call operator is interpreted by the right shell. Report upstream to GSD (`npx get-shit-done-cc`) once Windex post-merge dust settles.

- **Season-rollover over-projects future years (low priority, cosmetic):** `ensure_next_season_for_group()` (migration 021) is supposed to keep exactly one future season ahead of each group, but the idempotency check is "does a season starting on `<recent_end + 1 day>` already exist?" — and `recent_end` is recomputed each daily run from the *most recent* end_date. So once 2027 exists, the next day's run sees 2027's end_date as the new "recent" and creates 2028; the day after, 2029; and so on. Observed 2026-05-11: both Windex Cup and YC Windex have seasons 2027 / 2028 / 2029 already created instead of the intended single year ahead. Not user-visible — `windex-expo/lib/api.ts listSeasons` filters out `start_date > today`, so future rows never reach the picker. Fix: change the function to bail out once any season with `start_date > today` exists for the group, so the rollover only fires once we're within a calendar year of needing the next one. Until fixed, the daily cron will continue to add one more future row per day until manually trimmed.

- **Leap-year drift in season-rollover dates (low priority, cosmetic):** `ensure_next_season_for_group()` (migration 021) computes `next_end = next_start + INTERVAL '1 year' - INTERVAL '1 day'`. When `next_start` is March 1 of a year following a leap year (or any date crossing a Feb-29 boundary), Postgres `+ INTERVAL '1 year'` truncates to the last valid day of February, which makes the resulting `end_date` drift back one day. For golf league seasons this is invisible — Windex Cup's Dec-Nov and YC Windex's Sept-Aug never cross Feb-29 in their start_date. Flagged in case Windex ever picks up a group whose start_date IS in March, in which case we'd want to switch to month-based arithmetic instead.
